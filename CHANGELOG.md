### Added

- [#295] ADR-028 Addendum 1: Dynamic port override mechanism and GUI consequences. Documentation-only Addendum that introduces a minimal dynamic-port mechanism on the `Block` ABC (two override methods `get_effective_input_ports()` / `get_effective_output_ports()` + an enum-only declarative `dynamic_ports: ClassVar[dict | None]` mapping), locks the `BlockNode.tsx` GUI hardcoding removal (three `blockType === "io_block"` checks generalised to `category === "io"` plus Browse button generalisation), and **supersedes ADR-028 §D3** ("Seven core loader/saver pairs"): the six concrete loader/saver classes become two dynamic blocks `LoadData` and `SaveData` with twelve private module-level dispatch functions (`_load_array`, `_load_dataframe`, `_load_series`, `_load_text`, `_load_artifact`, `_load_composite_data`, and the symmetric `_save_*` set) in `load_data.py` / `save_data.py`. The user explicitly chose private functions over helper classes (Q-B) and enum-only mapping over a mini-DSL (Q-A). AI variadic port count and CodeBlock variadic port count are explicitly out of scope and deferred to ADR-029 (preliminary). Nine discussion-point decisions D1'–D9' (with prime marks to distinguish from ADR-028's D1–D14), four rejected alternatives (no dynamic mechanism, hidden classes with meta-palette, Pydantic discriminated union, combined variadic in same addendum). Detailed impact scope enumerates 7 backend files to modify, 4 backend files to create, 2 frontend files to modify, 5 test files to create, and an explicit "not affected" list covering Phase 10 core types / engine / other block bases / ADR-027 Addendum 1 worker reconstruction contract. Framework read-path migration: `Block.validate()`, `ProcessBlock.run()`, `workflow/validator.py`, `api/routes/blocks.py` all switch from `type(block).input_ports` to `block.get_effective_*_ports()`. Plugin IO blocks (`LoadImage`, `LoadMSRawFile`, etc.) remain static and get port colours via the existing `resolveTypeColor()` type-hierarchy walk — no per-plugin frontend work. 591 lines appended to docs/adr/ADR.md. - [#293] ADR-028: IOBlock architectural refactor — plugin-owned IO pattern. Documentation-only ADR that records the Phase 11 decision to delete the central adapter registry and refactor `IOBlock` into an abstract base class whose concrete subclasses are per-type loaders/savers. Core ships seven concrete loader/saver pairs (Array/DataFrame/Series/Text/Artifact/CompositeData; bare DataObject has no loader). The `scieasy.adapters` entry-point group is deleted (ADR-025 §6 superseded); plugins register IO blocks via the existing `scieasy.blocks` entry-point group. Pickle support is first-class but opt-in via `allow_pickle: bool = False` on LoadDataFrame/LoadSeries/LoadArray with an explicit security warning. Additional first-class format paths added: `.tsv`, `.pkl`, `.pickle`, `.npy`, `.npz`, `.parquet` for single-column Series. 14 discussion-point decisions D1–D14 plus four rejected alternatives (retrofit adapter with type_chain, subclass_of ClassVar, gradual deprecation, unified TypeRegistry-reader graph). Detailed impact scope enumerates 13 files to delete, 3 files to rewrite, 14 files to create, 9 documentation files to update in follow-up PRs, and explicit "not affected" list covering core types / engine / other block bases. Migration guide for workflow YAML files (`IOBlock(format=...)` → specific loader blocks). Enables Phase 11 Track 2 (scieasy-blocks-imaging), Track 3 (scieasy-blocks-srs), Track 4 (scieasy-blocks-lcms). 752 lines appended to docs/adr/ADR.md. - [#284] T-011: scieasy.utils.axis_iter.iterate_over_axes(source, operates_on, func) utility per ADR-027 D3. Iterates user-provided func over all axes NOT in operates_on, calls func(slice_data, coord_dict) for each combination, stacks results back into a new Array of source's concrete class. Memory: O(one input slice + one output slice). Serial by design (no threads, no subprocesses) per ADR-027 D8. Metadata propagation: framework derived (with `derived_from=source.framework.object_id`), meta shared by reference, user shallow copy, axes preserved. Raises `BroadcastError` when operates_on is not a subset of source.axes, when source.shape is None, when slice outputs have inconsistent shapes, or when func returns arrays with a different number of dimensions than len(operates_on); user-raised errors propagate unchanged. Imports `BroadcastError` from the sibling `scieasy.utils.broadcast` module (ADR-027 D3 referenced `scieasy.core.exceptions` which does not yet exist in the repo) so one `except BroadcastError:` clause covers both utilities. Sibling of `broadcast_apply` for the "single Array, iterate extra dims" pattern. Replaces the T-011 `NotImplementedError` skeleton from PR #262; removes the corresponding smoke tests from `tests/core/test_phase10_skeleton.py`; adds 16 per-feature tests in `tests/utils/test_axis_iter.py` covering the ADR-027 D3 acceptance criteria plus edge cases (empty extra axes, func reducing dim count, subclass preservation, metadata propagation, user-func error propagation).
